# Dagster & dbt Integration Guide

This document explains how Dagster and dbt work together in this project to create a seamless, asset-based data platform.

## 1. Roles & Responsibilities

| Feature | dbt (The "Transform") | Dagster (The "Orchestrate") |
| :--- | :--- | :--- |
| **Logic** | Defines **SQL** transformations (`SELECT ...`). | Defines **Python** assets (`@dbt_assets`). |
| **Execution** | Compiles Jinja to SQL and runs it against DB. | Triggers the dbt command (`dbt build`) at the right time. |
| **Dependencies** | Manages internal model dependencies (`ref()`). | Manages cross-stack dependencies (e.g., "Run dbt only after data arrives in S3"). |
| **Observation** | Generates documentation and basic tests. | Provides lineage, run history, asset health, and freshness SLAs. |

## 2. How Integration Works (`dagster-dbt`)

We use the `dagster-dbt` library, which treats dbt models as **Software-Defined Assets**.

### The Workflow
1.  **Parse**: At code load time, Dagster parses the `manifest.json` generated by dbt.
2.  **Asset Generation**: Every dbt model (`stg_orders`, `fact_sales`) automatically becomes a Dagster Asset in the UI.
3.  **Execution**: When you materialize an asset in Dagster, it translates that request into a specific dbt command (e.g., `dbt build --select fact_daily_sales`).

### Critical Configuration Files

#### `dbt_project/dbt_project.yml`
*   Defines the dbt project structure.
*   **Key Config**: `models:` section defines materialization strategies (e.g., `incremental`) which dbt handles, but Dagster visualizes.

#### `dagster_project/assets/dbt_assets.py`
This is the bridge code.
```python
@dbt_assets(manifest=manifest_path)
def retail_dbt_assets(context: AssetExecutionContext, dbt: DbtCliResource):
    yield from dbt.cli(["build"], context=context).stream()
```
*   **`@dbt_assets`**: This decorator tells Dagster "Read the manifest and create assets for me."
*   **`dbt.cli(["build"])`**: This executes the dbt command when the asset is triggered.

#### `dagster_project/definitions.py`
*   **Resources**: Configures the `DbtCliResource` with the correct paths to the dbt project and executable.
*   **Sensors**: Defines *when* to run.

## 3. Automation: The Sensor Pattern

We don't just run dbt on a cron schedule. We use **Declarative Automation** (Sensors).

### The "Source Asset" Pattern
In `dagster_project/assets/source_assets.py`, we define the raw tables as `AssetSpec` (or `observable_source_asset`).
```python
raw_orders = AssetSpec(key="raw_orders", group_name="ingestion")
```
This tells Dagster: "There is a table called `raw_orders` that exists outside of your control, but you should watch it."

### The Automation Condition
In `definitions.py`, we attach an `AutomationConditionSensor`.
*   **Logic**: "If `raw_orders` is updated (new data detected), automatically trigger the downstream dbt models (`stg_orders` -> `fact_sales`)."
*   **Benefit**: The pipeline is **event-driven**. Data flows through the system as soon as it is ingested, minimizing latency.

## 4. Why this is "Better" than Airflow + dbt

In a traditional Airflow setup, you might have a DAG that runs `dbt run` every hour.
*   **Problem**: If `dbt run` fails, you just see "Task Failed". You don't know *which* model failed without digging into logs.
*   **Problem**: You run it every hour regardless of whether new data arrived (wasteful) or if data arrived 5 minutes after the hour (latency).

**With Dagster:**
1.  **Granular Visibility**: You see exactly which asset (`fact_daily_sales`) failed in the lineage graph.
2.  **Asset-Awareness**: You can "Materialize" just the failed asset and its downstream dependencies.
3.  **Event-Driven**: Transformations run only when necessary.
